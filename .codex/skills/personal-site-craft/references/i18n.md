# Localization

## Static Pages

Prefer static localized routes over query parameters:

- `/` for English or default;
- `/es/`, `/ca/`, `/zh/`, `/ru/`, etc.

Each localized page should have:

- correct `<html lang>`;
- localized title and description;
- canonical URL pointing to itself;
- full reciprocal `hreflang` set, including `x-default`;
- localized UI labels where appropriate;
- localized OG/Twitter descriptions and image alt text;
- localized JSON-LD `ProfilePage.description`, `Person.name` where natural, and project descriptions.

## Language Detection

Client-side language detection is acceptable for a static site if:

- it only redirects from `/` or `/index.html`;
- it respects an explicit language chosen by the user;
- it does not redirect from localized URLs;
- it preserves search and hash.

Keep functional terminal hints such as `$ open terminal` in English if the UI concept is intentionally terminal-like.

## Translation Quality

Avoid mechanical translations. For each language:

- rewrite idiomatically instead of replacing words one by one;
- keep protocol/project names such as MCP, TC39, W3C, Test262, ECMA-262, OpenTelemetry;
- keep technical English only where a local equivalent would sound stranger to developers;
- avoid third-person-from-self phrasing on a personal homepage when it feels unnatural;
- read generated pages in context, not just locale tables.

Known preferences from this project:

- Russian: "разработчик" is often better than "инженер-программист" in casual/retro copy; "наблюдаемость" is better than raw "observability" in normal UI labels.
- Spanish/Catalan: `CV` is normal for compact link labels.
- Chinese: keep the name in English unless the user asks for a localized name; for Russian, localize the visible hero name to "Сергей Рубанов".

## Generated Locales

If localized pages are generated, never manually patch only `es/index.html`, `ru/index.html`, etc. Edit the generator and rebuild.
